01 / What It Is

The prototype in one paragraph

EKO-AI is a local, offline, zero-API-call knowledge assistant that answers questions from a set of company policy documents. You run a Python server on your laptop, open an HTML file in your browser, and ask questions. The system searches 24 approved policy documents covering all five company towers (AuthEnq, EIS, ITS, Digital, Operations) and returns a direct, sourced answer — citing the exact document ID and section — in under 2 milliseconds per query. Every question is logged. Questions the system can't find an approved source for are refused explicitly, not guessed at.

The one claim that matters for the pitch

After running pip install, disconnect your laptop from WiFi. The entire system — search, answers, audit log — still works perfectly. There is no external network call anywhere in the answer path. This is verifiable by anyone who reads the code, which is exactly the argument for a security-conscious organization.

What this prototype proves vs what it doesn't prove yet

Proven

Core architecture works

Retrieval, chunking, citation, audit log, refusal path, and the zero-API-call security design all work end-to-end at prototype scale.

Proven

Sub-2ms query latency

TF-IDF in-memory retrieval is measurably fast even on a laptop CPU with no GPU or special hardware.

Not yet proven

Scale beyond ~500 documents

The in-memory index works beautifully at 134 chunks / 27 documents. At 10,000+ chunks, you'd want a proper vector database (Qdrant, pgvector) — see the upgrade path section.

Not yet proven

Richer language understanding

TF-IDF matches vocabulary, not meaning. "Photo restriction" and "camera rule" are the same concept but different words — a neural embedding model handles this better. The upgrade path shows exactly where to add one.

03 / How It Works

The full pipeline, step by step

There are four stages that run once at startup, and then a fifth stage that runs every time someone asks a question.

Startup pipeline (runs once, ~300–500ms)

  1. Extract — extract.py reads every file in sample-docs/ and sample-images/, routing each file to the right reader based on its extension: pypdf for PDFs, python-docx for Word files, plain text reader for Markdown, and PIL image metadata for images. Output: one raw text string per document.
  2. Chunk — chunk.py splits each document into overlapping sections of ~220 words. It first tries to split on section headings ("## Heading" in Markdown, or "1. Section Title" in PDF-extracted text). For any section longer than 220 words, it applies a sliding-window sub-split with 40-word overlap. Each chunk is tagged with its document ID, title, section name, status ("approved"), and metadata. Output: 134 Chunk objects across all 27 documents.
  3. Build TF-IDF index — retrieval.py tokenizes every chunk's text, builds a vocabulary of 1,866 unique words across the corpus, computes IDF weights (rarer words get higher weight), and produces a normalized TF-IDF vector for each chunk. Vectors are stored as a NumPy matrix in memory. Output: an in-memory searchable index.

Per-question pipeline (runs in ~1ms)

  1. Vectorize the question — The user's question is tokenized using the same vocabulary built at startup, and converted into a TF-IDF vector using the same IDF weights. Words in the question that weren't in any document are simply ignored.
  2. Cosine similarity search — A single matrix-vector multiply scores the question against every chunk simultaneously. Chunks with status "approved" are the only ones eligible to score — draft or retired documents are excluded at the query level, not in a post-filter step. The top 5 chunks are returned.
  3. Answer synthesis — synthesize.py re-joins PDF-wrapped lines into whole sentences, strips numbered list markers, and ranks each sentence by how many query words it contains. The best 1–2 sentences from the top 1–3 chunks are combined into the final answer. Sources (doc_id, section, relevance score) are attached.
  4. Audit log — Before returning the response, audit.py appends one line to backend/data/audit_log.jsonl recording the timestamp, query, whether an answer was found, confidence level, which document IDs were used, and query latency in milliseconds.

TF-IDF in plain language

TF (Term Frequency) = how often a word appears in this chunk. IDF (Inverse Document Frequency) = how rare that word is across all chunks. A word like "gratuity" that appears in one chunk and nowhere else is a strong signal; a word like "policy" that's in every document tells you almost nothing. TF-IDF multiplies both together, so rare-but-locally-frequent words score highest — exactly the behaviour you'd want for finding a specific policy section.

Cosine similarity then measures how much a query's word-pattern overlaps with each chunk's word-pattern. It's fast (a matrix multiply in NumPy), requires no GPU, runs on any laptop CPU, and has no internet surface area at all.

04 / Running It

How to run this locally — complete steps

Requirements

RequirementDetails
Python 3.10 or newerDownload from python.org — free
8 GB RAM minimum16 GB comfortable; 8 GB works for this prototype
Any modern browserChrome, Firefox, Edge, Safari — no extensions needed
No GPU neededTF-IDF runs on CPU only; this prototype is designed for that
No API key neededIntentionally; zero external dependencies at runtime
  1. Open a terminal inside the backend folder. 
    cd eko-prototype/backend
  2. Install all Python dependencies. This only needs to happen once. All free No API keys 
    pip install -r requirements.txt
  3. Start the server. On startup, the server automatically reads every document in sample-docs/ and sample-images/, builds the index, and starts listening on port 8000. 
    uvicorn main:app --reload --port 8000
    Watch for this line in the terminal output — it means the index is built and the server is ready: 
    Index ready. Server is now accepting requests.
  4. Open the frontend. No build step — just open the file directly in your browser. 
    # On Mac:
open eko-prototype/frontend/index.html

# On Windows (run in cmd or PowerShell):
start eko-prototype/frontend/index.html

# Or just drag the file into any browser window.
  5. Ask your first question. Type in the chat input and press Enter. The "SYSTEM READY" status in the sidebar confirms the backend is connected. Answers typically appear in under 100ms from your keypress (dominated by browser rendering and network round-trip to localhost, not by search latency).
The --reload flag

--reload makes the server automatically restart whenever you edit a Python file — useful while developing. Remove it in any kind of stable/demo setup where you don't want accidental restarts.

05 / Testing

How to test that everything is working correctly

Quick sanity check — from the terminal

# Run this from the backend/ folder to test the full pipeline without starting the web server.
# It ingests all documents, builds the index, and runs a test search query, printing results.
python3 ingest.py

You should see every document listed with its chunk count, followed by a test search result showing the top-matching chunk for "can I take a photo of client code".

Interactive API explorer

FastAPI automatically generates interactive documentation at http://localhost:8000/docs — open this in a browser while the server is running and you can try every endpoint without writing any code.

06 / Adding Documents

How to add new documents and keep them searchable

  1. Drop the file into sample-docs/. Supported formats: .pdf, .docx, .md, .txt. For the document to be found accurately, structure it with numbered section headings like the existing docs — e.g. "## 1. Section Name" in Markdown, or "1. Section Name" as a line in a Word/PDF document.
  2. Add document metadata near the top of the file (optional but recommended). The ingestion pipeline looks for these lines in the first 10 lines of the file and uses them for document-level metadata and citation display: 
    Document ID: POL-XYZ-001
Owner: Team or Department Name
Status: Approved
Version: 1.0
Last Updated: 2026-06-22
  3. Trigger re-indexing — two options: 
    # Option A: call the reindex endpoint (no server restart needed)
curl -X POST http://localhost:8000/reindex

# Option B: restart the server (it re-reads everything on startup)
# Press Ctrl+C in the terminal where uvicorn is running, then run uvicorn again.
  4. Verify the new document appears in the document index at http://localhost:8000/documents and test a question about its content.
Important: never add real client data to this prototype

This prototype runs on your personal laptop and uses a simple local file for the audit log. Use only sample, invented, or public documents until this system is running on company-approved infrastructure with proper access controls.

07 / Upgrade Path

How this grows from prototype to production

Every upgrade below requires changing exactly one or two files — the rest of the pipeline stays identical. This is intentional: the stages were designed to be independently replaceable.

UpgradeChange whereWhat changes
Better language understanding (neural embeddings)retrieval.py — change the two marked functions in the "Upgrade Path" comment blockReplace TF-IDF vectorization with a locally-hosted model like all-MiniLM-L6-v2 via sentence-transformers. Requires a ~90MB model download. Everything else identical.
Persistent storage at scale (vector database)retrieval.py — swap TfidfIndex for a Qdrant or pgvector clientAt thousands of documents, in-memory storage becomes too large. A vector DB handles persistence and larger-scale retrieval. No change to extract, chunk, synthesize, or main.
Add a local LLM for richer answerssynthesize.py — add an Ollama API call after chunk selectionAfter retrieving the best chunks, pass them plus the query to a locally-hosted Llama 3 8B model to compose a smoother answer. The grounding constraint (answer only from the chunks) stays in the prompt. Adds latency; requires a GPU for acceptable speed.
Real authentication (SSO)main.py — add OAuth2/SAML middlewareReplace the current demo-user in the audit log with a real authenticated employee identity from the company SSO system.
Production audit databaseaudit.py — replace JSON-lines file with a PostgreSQL writeThe function signature stays identical; only the write destination changes. Enables proper querying, dashboards, and retention policies.
08 / Security Proof

Verifying the "no external call" claim

A security team reviewing this prototype can verify the zero-external-call claim in under ten minutes. Here is exactly what to look for:

  • No API key in any file. Search the entire codebase for "OPENAI_API_KEY", "ANTHROPIC_API_KEY", or any similar string. You will find zero results.
  • No requests or httpx import in the answer path. The only files that handle answering are retrieval.py, synthesize.py, and ingest.py — none of them import any HTTP client library.
  • No network socket opened during a query. Run the server and monitor network activity (e.g. tcpdump -i any port not 8000) while asking questions. You will see zero outbound connections.
  • The dependencies list confirms it. Every package in requirements.txt (fastapi, uvicorn, pypdf, python-docx, numpy, pillow) is purely local computation — none of them have cloud backends or phone-home behaviour.
  • The audit log is local. backend/data/audit_log.jsonl is written to disk on the same machine running the server. It goes nowhere else.
What this demonstrates for the enterprise proposal

This prototype proves all five core security claims from the EKO-AI proposal: (1) no external data transmission, (2) approved-only retrieval enforced at the query layer, (3) explicit refusal rather than fabrication, (4) every question logged immutably, and (5) source citation on every answer. All five are live, testable, working features — not planned roadmap items.
